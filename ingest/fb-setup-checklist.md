# Facebook-сбор — чек-лист настройки (Фаза 3, платно)

Главный источник, самый хрупкий и рискованный. Ключевой принцип: **не самопис, а managed-актор**
(Apify догоняет изменения Meta за тебя). Telegram держим как надёжный костяк — даже когда FB икнёт,
лента не пустеет. Юридика — `../docs/legal-pdpa.md` (храним факты+ссылку, без PII/ре-хостинга фото).

## 0. Заранее
- [ ] Решить бюджет (~$60-110/мес: Apify + резидентный прокси). Это цена FB-как-главного — Владимир согласен.
- [ ] Найти 3-5 ключевых FB-групп аренды Пхангана (часть закрытые). Записать их URL.

## 1. «Расходный» FB-аккаунт (не личный!)
- [ ] Завести отдельный FB-аккаунт **не на личные/рабочие данные RW**. Бан этого аккаунта не должен ничего ломать.
- [ ] Прогреть: аватар, имя, несколько дней нормальной активности, постепенно вступить в группы аренды Пхангана.
- [ ] Завести запасной аккаунт на случай checkpoint/бана.
- [ ] Хранить cookie-сессию аккаунта в `cookies/` (в `.gitignore`, в репозиторий НЕ коммитить).

## 2. Резидентный прокси (тайский IP)
- [ ] Выбрать провайдера: Bright Data / IPRoyal / Smartproxy (~$30-50/мес). Тайский резидентный IP, чтобы трафик выглядел локальным.
- [ ] Для list-only сбора (только текст, без фото) трафик лёгкий — большой объём GB не нужен.

## 3. Apify-актор
- [ ] Регистрация на Apify, тариф ~$30-60/мес (compute units + прокси отдельно).
- [ ] **Публичные группы + Marketplace-поиск «Koh Phangan»** (logged-off, риск ниже) — начать отсюда:
  - `apify/facebook-groups-scraper` (публичные группы) или `curious_coder/facebook-marketplace`.
- [ ] **Закрытые группы** (там много объектов) — актору подсунуть cookie «расходного» аккаунта + тайский прокси.
- [ ] Экономия/маскировка: `getAllListingPhotos=false`, `maxPagesPerUrl`, память 512МБ для list-only, встроенные задержки/ротация, низкая частота.
- [ ] Расписание: запуск каждые **3-6 часов** (это и есть «постоянный мониторинг»).

## 4. Стыковка с пайплайном
- [ ] Выход актора → нормализовать в формат `raw_post` (`../db/schema.sql`): source=`fb_group`/`fb_marketplace`, `source_url`, `external_id`, `raw_text`.
- [ ] Дальше — общий путь: Claude-парсер (`../parser/parse.py`) → нормализация района → дедуп → база.
- [ ] **Парсер устойчивый к поломкам:** Meta меняет разметку раз в 3-6 недель; парсер терпим к пустым полям.

## 5. Мониторинг «сломалось»
- [ ] Зафиксировать конкретный актор + версию (напр. `apify/facebook-groups-scraper` v1.2+) — чтобы знать, что именно сломалось при апдейте Meta.
- [ ] Если прогон актора вернул 0 объявлений или мусор → пуш тебе в Telegram «FB сломался».
- [ ] **Алерт на устаревание:** если последнего успешного прогона нет >14 дней → пуш «FB-ветка молчит». Meta меняет разметку каждые 2-4 недели, авторы акторов чинят с лагом.
- [ ] Закладывать: раз в пару месяцев FB-ветку придётся чинить/перенастраивать. Это нормально.

## 6. Юридическая гигиена (Thai PDPA)
- [ ] В базе — только фильтруемые факты + ссылка на оригинал. Телефоны/имена/фото НЕ хранить структурно.
- [ ] Privacy-notice на сайте, кнопка «убрать объявление», удаление по запросу ≤30 дней.
- [ ] Перед запуском автосбора — короткая консультация с тайским юристом по PDPA (база «legitimate interest»).

## Что НЕ делать
- ❌ Самописный скрейпер на голом Playwright как основной (вечная гонка с Meta — съест всё время соло).
- ❌ Личный/рабочий FB-аккаунт RW для скрейпа (риск потерять именно его).
- ❌ Импорт RW-инструмента `fb_bulk_*.html` — это инвентарь RW, не внешние объявления (см. `../docs/roadmap.md`).
